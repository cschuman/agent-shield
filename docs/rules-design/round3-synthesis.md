## 0. The meta-question Round 2 forced open

Critic A (strategic) made the strongest case in this whole exercise: **don't refactor yet.** Their argument, in one paragraph: Semgrep, Trivy, Falco all crystallized rule formats *after* commercial validation. Agent-shield has no paying customers, no named GRC partner, no GitHub issues asking "how do I add a framework," and zero outside contributors. The full Round 1 plan is realistically 6–8 weeks of sole-maintainer time, which is the same window in which the dashboard MVP could ship and produce revenue. A v1 schema designed pre-PMF will be wrong, and breaking it later is expensive forever.

Critic A is right that the timing question cannot be skipped. So Round 3 produces **three options**, not one:

| Path | Effort | When it's right | Risk |
|---|---|---|---|
| **A. Defer** — ship a 5–7 day internal Rust seam (extract `Rule` trait, unify guardrail/permission detection, write `CONTRIBUTING.md` for the Rust path, pin a tracking issue). No YAML. | ~1 week | If the next 90 days are about *getting customers and the dashboard*. If contribution demand is currently zero. | You stay the bottleneck on rules. Fine if rules-volume is low. |
| **B. Narrow refactor** — ship the YAML engine + first-party rules only. No external rule repos, no signing, no community PRs accepted yet. Rules-as-data is for *internal velocity and dogfooding*, not yet for community. | ~3 weeks | If you want the architecture in place before customers arrive, and you're willing to spend the time. | Schema lock-in starts. Mitigated because rules are still your own — you can refactor freely. |
| **C. Full refactor** — Round 1 as written, with Round 2 fixes baked in. External rule repos supported. Community PRs accepted. Signing infrastructure. | ~6–8 weeks | Only if a real signal triggers (see triggers below). | Premature platform-building. Schema you can't break for years. |

**Round 3's recommendation: Path B.** Not because Critic A is wrong about the timing — they're right that there's no community pull yet — but because the *internal* benefits of YAML rules (faster maintainer iteration, easier testing via fixtures, cleaner separation of concerns) accrue even with zero external contributors. You eat your own dog food; when contribution demand arrives, the schema is already battle-tested on your own rule set.

Path B explicitly **does not advertise the schema as stable** until one of these signals fires:

- 5+ outside contributors open issues asking how to add a framework
- A named GRC/CISO partner asks for control-mapping authoring rights
- A foundation, regulator, or insurer asks about adopting the format
- You ship 30+ first-party rules and the YAML pain is gone

Until then, the schema is documented as `experimental`, the README says "rules are data, but the format may change without notice," and you skip the signing/external-repo work entirely.

If you pick **Path A**, stop reading here. The rest of the document only matters for B or C.

## 1. What gets locked in (resolved Round 2 contradictions)

These are the architectural decisions the debate forced. Critic B (coherence) found ten conflicts; the consequential ones get resolved here.

### 1.1 `registry.yaml` is generated, not authoritative

A rule file is the single source of truth: detection + scoring adjustment + compliance mappings, all in one YAML. The `registry.yaml` index that backend-architect proposed exists, but it's a **build artifact** generated by CI from the rule files. Loaders never trust the registry over the rule. Frameworks/ is the canonical directory; compliance/ is a generated symlink view.

*Why:* security-auditor's single-file rule design wins on contributor velocity. Backend-architect's index is still useful for fast tag queries and checksum verification — it just isn't authoritative.

### 1.2 Closed `Matcher` enum, with a tiered contributor promise

rust-pro's "closed enum forever" wins. New matcher kinds require a Rust PR + binary release. But the contributor promise is **tiered explicitly**:

- **Tier 1 (text-level detection)** — `import_contains`, `code_regex`, `package_dep`, `file_present`, `multiline_regex`, plus `all_of`/`any_of`/`not` combinators. Adding a rule using only Tier 1 is the 30-minute path.
- **Tier 2 (context signals)** — fixed list shipped in v1.0: `tool_count`, `has_system_prompt`, `autonomy_tier`, `has_guardrail`, `has_permission`, `data_source_count`. Rules can reference these via `context_signal:`. New context signals are explicitly out of scope for community contribution; they require an RFC + Rust PR.
- **Tier 3 (new matcher primitives)** — closed. Only the maintainer ships these.

The README and CONTRIBUTING.md must say this plainly: *"Detection by text patterns is open. Detection by behavioral inference is, for now, gated."*

This kills the dx-optimizer/rust-pro contradiction that Critic B flagged as the highest-risk item in the whole design.

### 1.3 Reserve `extends:` from day one (Critic C's decisive call)

Critic C's two-year forensic identified the single most expensive deferred decision: not reserving an overlay/extension mechanism in v1. Their incident anchor was a GRC analyst trying to backfill EU AI Act mappings across 23 rule files in 2027 — a job that should have been a single overlay file.

**Decision:** the rule schema reserves an `extends:` top-level key in v1.0 even though the engine doesn't process it yet. The field is documented as "reserved; ignored in this version; will support overlay rules in v1.1+." This costs nothing today and prevents the most likely v2 break.

A Round-3-only addition to the schema:

```yaml
schema_version: "1.0"
extends: null   # reserved; null in v1.0; future: rule-id of the rule being overlaid
```

Schema validation rejects any non-null `extends:` value in v1.0 with a clear error. When v1.1 arrives, existing rules need no migration.

### 1.4 Bundled-primary distribution; auto-signing only at release

backend-architect's bundled-rules + optional `update-rules` model wins. The CLI works fully offline with `include_bytes!()`-embedded defaults. Signing (minisign) is *automatic at bundle release time*, not at PR merge time — contributors never touch a signing key. This was Critic B's resolution to the signing-vs-PR-velocity contradiction.

For Path B (no community PRs yet), signing is deferred entirely until external rule repos are supported. The build-time embed is enough.

### 1.5 Per-rule `schema_version` + CLI's `RULE_SCHEMA_VERSION` constant

Rules whose `schema_version` exceeds what the CLI knows are *quarantined with a diagnostic* — never silently dropped, never crash the scan. This is rust-pro's quarantine pattern applied to versioning.

### 1.6 Rule ID format: `{framework}/{category}/{slug}`

Examples: `framework/langchain/agent-detected`, `behavior/permissions/system-command-access`, `compliance/nist/govern-1-3-no-oversight`. CI enforces uniqueness across the entire bundled set. Critic B flagged that no expert had specified this; it has to be specified.

### 1.7 Framework baselines live in `rules/frameworks/{name}.yaml`

Not in detection rules. A rule references the framework by ID and only emits `score_adjustment`. Changing a framework baseline (e.g. moving CrewAI from 50 to 55) requires a PR with written justification — the productive friction dx-optimizer wanted, applied to the smallest possible governance surface.

## 2. The MVP rule shape (Path B v1.0)

Concrete, decision-ready. All resolutions from §1 baked in.

```yaml
schema_version: "1.0"
extends: null   # reserved for v1.1+

id: "behavior/permissions/system-command-access"
name: "Agent has system command execution access"
description: >
  The agent can execute system commands via subprocess, os.system, child_process,
  or equivalent. This is the highest single-factor risk multiplier in the OWASP
  Agentic Top 10 and triggers Article 14 EU AI Act human-oversight requirements.
severity: critical
category: PermissionEscalation
tags: [signal:permission, tier:behavior]

when:
  any_of:
    - code_regex: 'subprocess\.(run|call|Popen|check_output)'
      languages: [python]
    - code_regex: 'os\.system\s*\('
      languages: [python]
    - code_regex: '(child_process|execSync|spawn)\s*\('
      languages: [javascript, typescript]
    - code_regex: 'Command::new'
      languages: [rust]
    - code_regex: 'exec\.Command'
      languages: [go]

score_adjustment: 20

compliance:
  nist_ai_rmf:
    - { control: "MANAGE 2.1", description: "Detect and respond to AI risks in deployment" }
  iso_42001:
    - { control: "A.8.4", description: "Human oversight for high-impact AI actions" }
  eu_ai_act:
    - { control: "Article 14", description: "Human oversight requirement" }
  owasp_agentic:
    - { control: "A01", description: "Excessive Agency" }

remediation: >
  Wrap system command execution behind a human-approval gate or remove the
  capability entirely. If commands must be executed, restrict to an allowlist
  and log every invocation with the originating prompt.
```

Every rule fits this shape. Three rule kinds will exist:
1. **Framework detection** (e.g. `framework/langchain/agent-detected`) — `score_adjustment: 0`, scoring comes from the framework catalog baseline.
2. **Behavioral** (e.g. the example above) — `score_adjustment > 0`, applies on top of the framework baseline.
3. **Mitigating** (e.g. `behavior/guardrails/human-approval-detected`) — `score_adjustment < 0`, capped per the existing -25 cap rule.

Framework catalog (`rules/frameworks/langchain.yaml`):

```yaml
schema_version: "1.0"
id: langchain
name: "LangChain"
baseline: 40
notes: "Risk floor for any code containing LangChain agent constructs."
references:
  - "https://python.langchain.com/docs/security"
```

- `rules/compliance/` is **not** a directory yet. The compliance dimension is queried from rule metadata, not separately filed. If we later want a compliance namespace, it's a CI-generated symlink view — never authored directly.
- `rules/registry.yaml` is **not** committed to the repo; CI generates it on release.
- Bundled rules are `include_bytes!()`'d at compile time. There is no runtime download of rules in v1.0.

## 4. Phasing (Path B, ~3 weeks of sole-maintainer time)

| Week | Deliverable | Reversible? |
|---|---|---|
| **Week 1** | Internal Rust seam: extract `Engine`, `Matcher` enum, `signals.rs`. Behavior of CLI is byte-identical to today. No YAML yet. Tests prove no regression. | Yes — pure refactor |
| **Week 2** | YAML loader. Migrate the existing 10 frameworks + the cross-cutting behavioral rules into `rules/*.yaml`. Hardcoded `frameworks.rs` deleted. CI enforces schema validation + regex compile + corpus run. | Mostly — rules are now data but private. Schema marked *experimental*. |
| **Week 3** | Quality of life: `agent-shield rule lint`, `agent-shield rule new` scaffold, fixture format documented. README updated. CONTRIBUTING.md says "schema experimental — internal use." | Yes — these are tools, not commitments |

Defer to Path C when triggered:
- External rule-repo support (`--rules <url>`)
- Minisign signing
- `update-rules` command
- CONTRIBUTING.md flip from "experimental" to "stable"
- Community PR templates and the productive-friction governance dx-optimizer designed

## 5. What we are *not* committing to in v1.0

- AST-based detection (Semgrep-style). Closed for v1.0; potentially revisited at v2.
- WASM/Lua/Rego pluggability. Closed forever per rust-pro's argument.
- Multiple rule schemas coexisting. v1.0 is one shape.
- Rule overlays (`extends:` semantics). Reserved syntactically, not implemented.
- Per-rule digital signatures (vs. bundle-level). Not a v1 concern.
- Auto-update on first run. Bundled-only.

## 6. Genuinely open questions Round 3 cannot resolve alone

These need either user input, real-world data, or a deferred decision:

1. **Path A vs. B vs. C choice.** Owner: user. Round 3 recommends B with explicit triggers to escalate to C.
2. **Whether to open the GitHub repo at all in Path B.** If rules are private/experimental, the repo can stay local until a customer or partner appears. Tradeoff: no SEO, no inbound interest. Recommend pushing public anyway with a clear "not seeking contributions yet" note in CONTRIBUTING.md.
3. **The framework-catalog governance.** Who can change a baseline? Today: just the maintainer. When a GRC partner arrives, this is the first thing they'll want to influence. Defer until then.
4. **Performance ceiling.** Critic B and backend-architect both flagged 500+ rules as the unproven scaling point. Realistic v1.0 target: 30–50 rules. We won't hit the ceiling for a year. Re-evaluate when bundled rule count crosses 100.
5. **Whether `context_signal` v1 should include `data_classification` (PII, PHI, secrets).** Currently not in the list. Adding it later requires a Rust PR. Recommend: hold the line — `data_source_count` is enough for v1.
