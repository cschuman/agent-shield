CASE STUDY · CALIBRATION · 2026

What happens when you point Agent Shield at real code.

v0.1.0 ships a 6-fixture snapshot oracle that locks the byte-identical contract. Fixtures don't tell you whether the scanner is good — they tell you it hasn't drifted. So we ran v0.1.0 against five real-world open-source agent repositories and wrote down everything we found, including the parts that didn't go well.

Date: 2026-04-25 Scanner: v0.1.0 (commit 78da44a) Method: 5 OSS repos · 5 parallel structured assessments
Section 01 · Setup

The five repos

Five repositories chosen for framework diversity and for stress: a framework's own monorepo (langgraph), an examples repo (crewAI-examples), a reference implementation set (MCP servers), a popular custom-loop agent (open-interpreter), and an SDK whose codebase is itself a framework's source (vercel/ai). The last one was deliberately chosen as a worst-case probe — what does the scanner do when pointed at the framework's own implementation?

Two of these numbers are wrong before any deeper inspection. Open Interpreter is unambiguously an agent project — zero detections is a recall bug. And 1,291 detections in vercel/ai is the SDK detecting itself: the scanner has no awareness that the file it's reading IS the framework, not user code that uses it.

Section 03 · What Was Wrong

Five structural failures

1. Bare imports were treated as agents

The LangChain rule fired on any file containing import langchain. In a langchain monorepo every file imports langchain — checkpoint backends, type stubs, test fixtures, SDK clients. None of those are agents. Same problem with the LangGraph and CrewAI rules.

2. Three-character substring matchers

The Anthropic MCP rule looked for "mcp" as a 3-char substring + the word "import" anywhere on the line. It matched connect_mcp, disconnect_mcp, client_mcp_v1, and tmcp. The Vercel AI rule looked for "ai/" — which matched every cross-package import inside the SDK monorepo (@ai-sdk/openai, @ai-sdk/provider-utils). The AWS Bedrock rule's bedrock-agent regex matched the URL string bedrock-agent-runtime.us-west-2.amazonaws.com.

3. Test directories were scanned by default

143 of 385 langgraph detections (37%) came from tests/ paths — pytest fixtures and unit tests exercising the framework's own surface. Test code lives inside the repo's own trust boundary; a security-tooling user doesn't get incremental risk insight from agent-shaped strings inside a vitest spec.

4. The Custom Agent rule didn't recognize real custom agents

Open Interpreter uses system_message (LiteLLM convention), not system_prompt. Its tool calls are dict-key form ("tool_call":), not assignment form. Its base model is the Anthropic SDK or LiteLLM, not the newer Anthropic Agent SDK. The rule was written against a too-narrow dialect and missed the most popular custom-agent project on GitHub entirely.

5. Score saturation

97.7% of vercel/ai detections were rated Critical. 91% of crewAI-examples scored exactly 100. The scoring rules apply every "missing guardrail" penalty to every detection regardless of detection confidence, so a 21-line console.log demo gets the same Critical rating as a production agent with shell access. When everything is Critical, nothing is.

Section 04 · What We Changed

Nine commits, one calibration round

Every fix was a tightening, not a structural rewrite. The 6-fixture snapshot contract held byte-identical through every commit. cargo test: 69 passing throughout. cargo clippy --all-targets -- -D warnings: clean throughout.

Rule-only fixes SHIPPED

  • LangChain, LangGraph, CrewAI: rewrite as all_of(import + positive constructor). Constructor allowlists: StateGraph, AgentExecutor, create_*_agent, Agent(, @CrewBase, etc. A bare import is no longer enough.
  • Anthropic MCP: drop the bare "mcp" substring. Require from mcp.<X> / import mcp.<X> or the @modelcontextprotocol SDK namespace.
  • Vercel AI: drop the bare "ai/" substring. Require all_of(import-from-'ai' + generateText|streamText|tool function-call).
  • AWS Bedrock: drop the bare bedrock-agent regex. Require anchored SDK class names (BedrockAgentClient, BedrockAgentRuntimeClient) or the @aws-sdk/client-bedrock-agent package.
  • Custom Agent: broaden the alternatives to recognize system_message (LiteLLM), dict-form "tool_calls": keys, OpenAI message-format {"role": "system"}, and litellm.completion() / Anthropic().messages.create() call sites.

Scanner change SHIPPED

  • Skip test directories by default. tests, test, __tests__, __testfixtures__, __mocks__, spec added to SKIP_DIRS. Six lines of code, ~37% reduction on the largest repo.

Architectural backlog DEFERRED TO v0.2

  • is_framework_self context signal. Read the repo's root package.json / pyproject.toml; if the package name matches a known framework, suppress that framework's own rule. Touches signals.rs, matcher.rs, and loader.rs in lockstep.
  • Manifest-pass activation. Matcher::matches_repo is wired but the v0.1 scanner only invokes matches_file. package_dep: and file_present: matchers are dormant until this lands.
  • Confidence-gate the scoring rules. Today every detection accumulates every penalty. A low-confidence match shouldn't paint the same Critical color as a high-confidence one.
  • .ipynb notebook parsing. The most agent-dense part of any LangGraph repo is its tutorial notebooks. Today the scanner is blind to them.
  • Python docstring stripping. Triple-quoted docstrings containing example code match real imports.

The vercel/ai number is not a failure. The 1,140 residual detections are correctly identifying SDK usage in framework-shipped example files. Those files genuinely call generateText() from the 'ai' SDK. Eliminating them isn't a rule-tuning problem — it's the framework-self-detection signal we deferred to v0.2. Until that lands, the honest answer is that running Agent Shield against vercel/ai is outside the intended use case: the scanner is for application repos that use a framework, not for the framework's own implementation.

Section 06 · Verdict

What we learned

The v0.1.0 detection rules were over-tuned to small fixture corpora and broke catastrophically on framework monorepos. Sample precision on langgraph and vercel/ai was 0/15 in both cases. A single calibration round of rule-only changes brought sample precision into a defensible range on every repo except vercel/ai, where the residual is the kind of false-positive that no rule edit can fix — only architecture can.

The v0.1.0 contract is preserved: every snapshot fixture remains byte-identical, every test passes, clippy stays clean, MSRV stays at 1.88. The byte-identical contract held through nine rule commits because every change was a tightening — narrowing the conditions under which a rule fires — not a structural rewrite.

The scoring layer was deliberately not changed in this round. The current scoring rules saturate at Critical on every detection; that's a real bug, but the fix (gating the missing-* rules behind detection confidence) requires a new context signal and is its own architectural change. v0.2 work.

This calibration is itself the most important fixture the project has. The 6 small fixtures lock the byte-identical contract; the 5 real-world repos lock the rule-quality contract. Both will run on every future change.

Section 07 · Reproduce

Run it yourself

git clone https://github.com/cschuman/agent-shield
cd agent-shield
git checkout v0.1.0  # or `calibration-v0.1.0` for post-tuning
cargo build --release

mkdir -p calibration/repos calibration/scans
cd calibration/repos
git clone --depth=1 https://github.com/langchain-ai/langgraph.git
git clone --depth=1 https://github.com/crewAIInc/crewAI-examples.git
git clone --depth=1 https://github.com/modelcontextprotocol/servers.git
git clone --depth=1 https://github.com/OpenInterpreter/open-interpreter.git
git clone --depth=1 https://github.com/vercel/ai.git
cd ../..

for r in langgraph crewAI-examples servers open-interpreter ai; do
  ./target/release/agent-shield scan calibration/repos/$r \
    --format json -o calibration/scans/$r.json
done

Detection counts will drift over time as the upstream repos evolve. The full engineering report is in the repo at docs/case-study/2026-04-25-real-world-calibration.md.
